Pensamento Computacional
Decompor, abstrair, reconhecer padrões
▼

Pensamento computacional não é sinônimo de programar — é uma forma de resolver problemas que pode (e deve) ser praticada longe do computador. São quatro pilares que, juntos, formam a base de qualquer programador eficiente.

DecomposiçãoQuebrar problemas grandes em problemas menores e manejáveis. Exercício: "Como fazer um sanduíche?" — decompor em sub-tarefas cada vez menores até chegar em instruções atômicas que um computador poderia executar.
Reconhecimento de padrõesIdentificar repetições e similaridades entre problemas diferentes. "Esse problema é parecido com aquele que resolvi antes?" é a pergunta mais valiosa que um programador pode fazer.
AbstraçãoFocar no que importa, ignorar o que não importa. Uma função é abstração: quem chama não precisa saber como funciona por dentro. Modelar o mundo em estruturas que um programa possa manipular.
Algoritmos e fluxogramasSequências de passos bem definidas. Fluxograma como linguagem visual para descrever algoritmos. Símbolos: início/fim (oval), processo (retângulo), decisão (losango), I/O (paralelogramo).
🎯
Entregável: Fluxograma de uma rotina cotidiana sua (ex: decidir o que comer no almoço, escolher um filme para assistir) com pelo menos 2 pontos de decisão. Papel é suficiente.

Pseudocódigo & Estruturas de Controle
Lógica em português estruturado
▼

Pseudocódigo é a ponte entre o pensamento humano e o código de computador. Escrever pseudocódigo força você a pensar na lógica sem se preocupar com sintaxe — e depois, traduzir esse pseudocódigo para Python na Fase 4 se torna natural. As estruturas de controle são os blocos com que toda lógica é construída.

Variáveis e tipos de dadosContêineres nomeados para guardar valores. Tipos: número inteiro, número decimal, texto, booleano (verdadeiro/falso). Por que o tipo importa — você não pode somar um número com um texto sem conversão.
Condicionais: SE / SENÃOO "se isso, então aquilo" que dirige toda lógica. SE … ENTÃO … SENÃO … FIM-SE. Condições compostas: E (AND), OU (OR), NÃO (NOT). Tabela verdade de forma intuitiva.
Laços: ENQUANTO e PARARepetição até uma condição ser atendida (enquanto) vs. repetição um número fixo de vezes (para). Loop infinito: o que é e como evitar. Diferença entre iteração e recursão (prévia).
Funções (procedimentos)Blocos reutilizáveis de código com nome. Parâmetros de entrada e valor de retorno. O princípio DRY (Don't Repeat Yourself). Por que escrever a mesma lógica duas vezes é sempre um sinal de que há um problema de design.
🎯
Entregável: Pseudocódigo de um algoritmo para verificar se um número é primo, com as estruturas de controle corretas. Você executa o algoritmo "na mão" com três exemplos (um número primo, um composto e o número 1) para validar.

Algoritmos Clássicos
Busca, ordenação e recursão
▼

Alguns problemas aparecem tão frequentemente na computação que soluções clássicas e eficientes foram desenvolvidas e estudadas por décadas. Conhecê-las não é só saber a solução — é desenvolver o vocabulário e a intuição para reconhecer quando um problema se encaixa em um padrão conhecido.

Busca linearVerificar elemento por elemento. O(n). Quando usar: listas pequenas ou não ordenadas. Simples mas ineficiente para grandes volumes.
Busca bináriaDividir pela metade a cada passo (precisa de lista ordenada). O(log n). A diferença entre buscar em 1 bilhão de itens com busca linear vs. binária: 1 bilhão de passos vs. ~30 passos.
Bubble Sort e Selection SortAlgoritmos de ordenação simples para visualizar o conceito. Não são eficientes (O(n²)) mas são fáceis de entender passo a passo. Visualizações animadas ajudam muito aqui.
Recursão — ideia geralUma função que chama a si mesma. O caso base (quando parar) e o caso recursivo. Fatorial e Fibonacci como exemplos clássicos. Pilha de chamadas e o risco de stack overflow.
Complexidade: O(n) intuitivoBig O como vocabulário para descrever eficiência. O(1), O(log n), O(n), O(n²). Não precisa de prova matemática — precisa da intuição: "se dobrar o input, o que acontece com o tempo?"
🎯
Entregável: Você implementa busca linear e busca binária em pseudocódigo e explica verbalmente por que a busca binária é mais eficiente — usando um exemplo concreto com números reais para demonstrar a diferença de passos.

Estruturas de Dados
Listas, pilhas, filas e dicionários
▼

Estruturas de dados são as formas de organizar informação em memória. A escolha da estrutura certa para um problema pode ser a diferença entre um programa eficiente e um lento. Esta etapa apresenta as estruturas fundamentais com analogias do mundo real antes de qualquer código.

Arrays / ListasSequência ordenada de elementos com índice numérico. Acesso por índice é O(1). Inserção no meio é custosa. Analogia: assentos numerados numa fileira de cinema.
Pilha (Stack)LIFO — Last In, First Out. Último a entrar, primeiro a sair. Analogia: pilha de pratos. Uso real: histórico de "desfazer" (Ctrl+Z), pilha de chamadas de funções.
Fila (Queue)FIFO — First In, First Out. Primeiro a entrar, primeiro a sair. Analogia: fila de banco. Uso real: fila de impressão, sistema de atendimento ao cliente.
Dicionário / Mapa (Hash Map)Pares chave-valor. Acesso por chave em O(1) médio. Analogia: dicionário de português — você busca pela palavra (chave) para encontrar a definição (valor). Uso real: quase tudo em programação moderna.
🎯
Entregável: Para cada estrutura, você inventa uma situação do cotidiano que seria modelada com aquela estrutura e explica por que escolheu ela e não outra. Quatro situações, quatro justificativas.

Código Limpo & Depuração
Escrever código que outros (e você mesmo) entendem
▼

Código que funciona é o mínimo. Código limpo é o que separa programadores iniciantes de profissionais. Esta etapa introduz os princípios de legibilidade, organização e depuração — habilidades que você vai usar pelo resto da carreira, independente da linguagem.

Nomes que comunicamVariáveis e funções com nomes descritivos. A regra: quem lê o nome deve entender o propósito sem ler a implementação. x vs. idade_usuario.
Comentários com propósitoComentar o "por quê", não o "o quê". "Somei 1 aqui" é inútil. "Adicionei 1 para compensar índice base-zero" é útil. Código bem escrito se explica sozinho na maioria das vezes.
Funções com responsabilidade únicaPrincípio SRP: cada função faz uma coisa e faz bem. Funções longas são o sinal mais claro de que algo precisa ser refatorado. "Se você precisar usar 'e' para descrever o que a função faz, ela faz coisas demais."
Depuração sistemáticaDebugging não é adivinhar — é método. Leia o erro com cuidado (a mensagem geralmente diz onde está o problema). Use print() ou um debugger para inspecionar o estado. Reduza o problema: qual é o menor código que reproduz o bug?
🎯
Entregável: Você recebe um código intencionalmente mal escrito (variáveis com nomes sem sentido, funções longas, sem comentários) e o refatora: renomeia variáveis, quebra em funções menores e adiciona comentários onde necessário. O código final deve ser legível por outra pessoa sem explicação verbal.
