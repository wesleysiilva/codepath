Processos e memória:
malabarismo do SO

Você tem 20 abas no Chrome, música no Spotify e o VS Code aberto ao mesmo tempo. Uma única CPU. Como isso é possível? O sistema operacional faz um malabarismo sofisticado de processos, threads e memória — e esta aula explica como.

⏱ ~15 min
📶 Iniciante
Módulo 3 de 5

O que é um processo?

Um processo é um programa em execução. Não o arquivo do programa no disco — o programa rodando: com memória alocada, arquivos abertos, estado atual. Cada processo tem um identificador único chamado PID (Process ID).

Abrir o Chrome cria um processo. Abrir uma segunda janela pode criar outro. Executar python script.py cria um processo Python. Cada um tem sua própria área de memória — isolada dos outros.

chrome.exe PID 4312
1.2 GB
spotify.exe PID 2891
340 MB
Code.exe PID 7043
410 MB
python.exe PID 9120
90 MB

Na fila, aguardando a CPU

O processo está carregado na memória e pronto para executar — mas a CPU está ocupada com outro processo. O escalonador (scheduler) decide quem roda a seguir.

Rodando (Running)

Usando a CPU agora

O processo está ativamente usando um núcleo da CPU. Em um processador com 8 núcleos, até 8 processos podem estar rodando simultaneamente (um por núcleo).

Esperando (Waiting)

Bloqueado aguardando I/O

O processo pediu uma operação lenta — ler um arquivo, esperar resposta da rede — e ficou bloqueado. Libera a CPU para outros processos enquanto espera. É aqui que o multitasking ganha eficiência real.

Scheduling: como o SO divide a CPU

Com dezenas de processos querendo usar a CPU, o escalonador (scheduler) do kernel decide quem roda e por quanto tempo. A técnica principal é o time slicing: cada processo recebe uma fatia de tempo (tipicamente 10–20ms) antes de ser pausado e o próximo assumir.

🎰
Concorrência vs. paralelismo: Com 1 núcleo, o SO cria a ilusão de múltiplos programas simultâneos alternando muito rápido (concorrência). Com múltiplos núcleos, processos realmente rodam ao mesmo tempo (paralelismo). Seu computador provavelmente tem ambos: paralelismo real entre núcleos e concorrência dentro de cada núcleo.

Threads: divisão dentro do processo

Um processo pode ter múltiplas threads — fluxos de execução que compartilham a memória do mesmo processo. Se um processo é uma fábrica, threads são os trabalhadores dentro dela.

Processo

Unidade de isolamento

Memória própria, arquivos próprios, PID próprio. Processos diferentes não veem a memória um do outro. Criar um processo é caro (copia toda a memória).

Thread

Unidade de execução

Compartilha a memória do processo. Criar uma thread é barato. Threads do mesmo processo podem se comunicar diretamente — mas isso exige cuidado para evitar condições de corrida (race conditions).

O Chrome usa múltiplos processos (uma aba travada não derruba as outras) e múltiplas threads por processo (renderização, JavaScript e I/O em paralelo). Quando você usa threading ou asyncio em Python, está trabalhando diretamente com esse modelo.

Memória virtual: a ilusão da memória infinita

Cada processo acredita que tem acesso exclusivo a um grande espaço de memória — como se fosse o único programa rodando. Isso é a memória virtual. O kernel mapeia esses endereços virtuais para a RAM física real (e para o disco quando necessário).

  1. 1
    Isolamento: o processo A não consegue acessar a memória do processo B — o kernel garante isso. Um programa com bug não pode corromper outro.
  2. 2
    Swap: quando a RAM enche, o kernel move páginas de memória menos usadas para o disco (arquivo de paginação). O processo não sabe — mas fica lento. É por isso que o PC trava quando a memória esgota.
  3. 3
    Compartilhamento: código de bibliotecas (como a libc) pode ser mapeado na memória de vários processos ao mesmo tempo — economizando RAM real.
💥
Segmentation fault — o erro de memória: Quando um programa tenta acessar um endereço de memória que não foi alocado para ele (ou que pertence ao kernel), o SO encerra o processo imediatamente com "Segmentation fault". É uma proteção — sem ela, o programa poderia corromper dados de outros processos. Em Python você raramente vê isso; em C e C++, é um erro clássico.

Termos-chave desta aula

Processo Instância de um programa em execução. Tem PID, memória própria, arquivos abertos e estado. Isolado dos outros processos pelo kernel.
PID Process ID. Número único que identifica um processo no sistema. Visível no Gerenciador de Tarefas (Windows) ou com ps / top (Linux/Mac).
Thread Fluxo de execução dentro de um processo. Compartilha a memória do processo. Mais leve de criar que um processo, mas exige sincronização cuidadosa.
Scheduler Componente do kernel que decide qual processo/thread usa a CPU em cada momento e por quanto tempo (time slice).
Memória virtual Abstração que dá a cada processo a ilusão de ter um espaço de memória exclusivo e contíguo. Mapeada para RAM real (e swap no disco) pelo kernel.
Swap Espaço em disco usado como extensão da RAM quando esta esgota. Muito mais lento — é o que causa lentidão quando o sistema "trava por falta de memória".
